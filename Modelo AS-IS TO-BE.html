El Problema

1. MODELO AS-IS (Situación Actual)

1.1 Contexto del Problema

AGP es una compañía del sector automotriz que fabrica vidrio blindado. Su sede en Ciudad de México mantiene un stock de más de 3,800 piezas físicas, algunas con hasta 5 años de almacenamiento. El CEO exige una estrategia de reducción de inventario urgente.

3,838
Piezas en inventario
55%
Pedidos incompletos
8-10h
Horas semanales de trabajo manual
5 años
Máximo tiempo en bodega

1.2 Pain Points del Proceso Actual

🕐 Proceso 100% Manual

Búsqueda en Excel, validación por correo. 8-10 horas semanales por ingeniero comercial.

❌ Sin Priorización por Antigüedad

Piezas con 5 años esperando junto a piezas nuevas. No se priorizan las más antiguas.

📧 Validación por Correo

Confirmación con partners logísticos toma días. Sin respuesta inmediata.

📝 Sin Trazabilidad

No hay registro de qué se emparejó, cuándo, ni por quién. Sin auditoría.

⚠️ Errores Humanos

Emparejamientos incorrectos por fatiga, descuido o desconocimiento de reglas.

📈 No Escalable

Más pedidos = más horas linealmente. No hay eficiencia ganada con volumen.

La Solución

2. MODELO TO-BE (Propuesta con Tecnología)

2.1 Solución Propuesta

Una herramienta web que aplica automáticamente las tres reglas de negocio de AGP: mismo cliente, mismo vehículo, mismo producto — con priorización automática por antigüedad (siempre primero las piezas más antiguas).

2.3 Comparación AS-IS vs TO-BE

3.1 Arquitectura de la Solución

El sistema está diseñado con separación de responsabilidades para facilitar integración futura:

Componente Tecnología Responsabilidad
Interfaz Web Streamlit Carga de archivos, visualización de resultados, descarga de reportes
Motor de Reglas Python + Pandas Aplicación de reglas de negocio, priorización por antigüedad, generación de pares
Exportación Excel (openpyxl) 3 hojas: Pares, Sin Par, Datos Problemáticos

3.2 Decisiones de Arquitectura para Escalabilidad

La arquitectura modular fue elegida estratégicamente para permitir evolución sin reescritura:

Decisión Técnica Beneficio para Escalabilidad Ejemplo de Evolución
Separación Motor de Negocio / Interfaz El motor Python es independiente de Streamlit Se puede integrar con sistemas ERP sin modificar las reglas de emparejamiento
Entrada/Salida basada en DataFrames Cualquier fuente de datos estructurados es compatible Conexión directa a base de datos SQL, API REST, o servicios cloud
Reglas de negocio parametrizables Las 3 reglas están en funciones reutilizables Agregar nueva regla (ej: prioridad por tipo de cliente) sin refactoring
Exportación multi-formato Arquitectura permite múltiples salidas Email automático, notificaciones push, actualización directa en sistemas ERP
Cálculo estadístico desacoplado Métricas generadas como módulo separado Dashboard en tiempo real, reportes ejecutivos automáticos, KPIs en BI

Estado Actual: Prototipo funcional probado con los 3,838 registros reales de AGP. El sistema está listo para uso inmediato y preparado para crecer según las necesidades del negocio.
